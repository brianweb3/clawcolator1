Clawcolator Documentation

Complete guide to the agent-first perpetual futures protocol

Overview

Clawcolator is an agent-first fork of Percolator where all market decisions are delegated to an autonomous OpenClaw agent, while the protocol strictly enforces financial invariants and system safety.

Key Concept

Unlike traditional protocols where humans or formulas make decisions, Clawcolator centralizes all decision-making in the OpenClaw agent. The protocol layer only validates and enforces these decisions mathematically.

Core Principles

  • Decision Layer: OpenClaw agent thinks and chooses actions
  • Enforcement Layer: Protocol validates and executes with mathematical guarantees
  • Separation of Concerns: Agent can propose anything; protocol can only allow or reject

What Makes Clawcolator Different

Traditional Protocols Clawcolator
Multiple decision-makers (LP, matching, governance) Single autonomous agent (OpenClaw)
Market logic embedded in protocol Protocol has zero market logic
Complex coordination between systems Simple: agent proposes, protocol validates

Architecture

Clawcolator follows a two-layer architecture that cleanly separates decision-making from enforcement.

System Architecture

Data Flow

  1. OpenClaw agent analyzes market conditions and proposes actions
  2. Clawcolator Core receives proposals and validates them
  3. Risk Engine performs formal verification checks
  4. If valid, action is executed; if invalid, action is rejected

OpenClaw Agent

The OpenClaw agent is the sole decision-making entity in Clawcolator. It has complete autonomy over market operations while being constrained by protocol safety rules.

Agent Responsibilities

  • Trade Acceptance: Accepts or rejects trades based on market analysis
  • Liquidity Management: Determines active capital allocation and redistribution
  • Market Parameters: Dynamically sets leverage, position limits, spreads, and funding rates
  • Matching: Acts as counterparty and determines execution conditions
  • Risk Management: Reduces exposure, hedges, and closes positions when threatened
  • Anomaly Response: Reacts to volatility, oracle issues, and manipulation attempts
  • Emergency Shutdown: Can initiate controlled shutdown if market is unsafe
⚠️ Agent Limitations

The agent cannot directly access funds or bypass protocol checks. All actions must pass through the enforcement layer, which can reject any proposal that violates financial invariants.

Protocol Layer

The Clawcolator Core protocol layer contains zero market logic. It only validates and enforces agent decisions.

What the Protocol Does

  • Calculates balances and PnL
  • Checks margin requirements and liquidation eligibility
  • Monitors financial invariants
  • Rejects any agent action that would violate system safety

What the Protocol Does NOT Do

  • Make trading decisions
  • Set market parameters
  • Choose counterparties
  • Determine risk tolerance
✅ Safety Guarantee

No agent action can bankrupt the system, violate financial invariants, or lead to negative balances. The protocol mathematically guarantees this.

Safety Guarantees

Clawcolator provides formal mathematical guarantees about system safety and fund conservation.

Financial Invariants

  • Fund Conservation: Total withdrawable claims ≤ Vault balance
  • Principal Protection: One account's insolvency cannot reduce another's principal
  • Non-Negative Balances: No account can have negative capital
  • Isolation: User actions don't affect other users' protected principal

Formal Verification

The underlying Percolator risk engine is formally verified using Kani verifier, proving:

  • Conservation of funds across all operations
  • User isolation guarantees
  • No-teleport behavior for cross-LP closes

Risk Management

Clawcolator implements comprehensive risk management through the formally verified Percolator risk engine.

Margin Requirements

  • Initial Margin: Required for risk-increasing operations
  • Maintenance Margin: Minimum margin to keep positions open
  • Dynamic Adjustment: Agent can modify margins based on market conditions

Liquidation Process

  1. System detects account below maintenance margin
  2. Position is closed at oracle price
  3. Losses are realized and socialized if necessary
  4. Remaining capital is returned to user

Liquidity Management

The OpenClaw agent manages liquidity dynamically, allocating capital based on market conditions and risk assessment.

Liquidity Allocation

  • Agent determines active capital percentage
  • Funds are redistributed based on exposure needs
  • Protective mode activates when risk thresholds are exceeded

Matching Engine

The agent acts as the matching engine, determining execution conditions and acting as counterparty when needed.

Matching Process

  1. User proposes trade action
  2. Agent analyzes proposal and market conditions
  3. Agent decides to accept, reject, or modify
  4. If accepted, agent acts as counterparty or finds match
  5. Trade executes at agent-determined conditions
